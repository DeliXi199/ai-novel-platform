## 先给一句结论

这个项目**不是“发一个 prompt，拿回一章文”那么简单**。  
它本质上是一个“**数据库持久化 + Story Bible 运行时总状态 + 严格章节流水线 + 异步任务系统 + 静态前端工作台**”组成的小说生产平台。

最核心的一点是：

- **数据库表**负责保存“书、章、摘要、人物、干预、任务”这些稳定对象。
- **`Novel.story_bible` JSON**负责保存“规划、近纲、长期状态、事实账本、Story Workspace 快照、生成运行时状态”这些复杂结构。
- **章节生成流程**每生成一章，都会一边写正文，一边把新的事实、人物状态、伏笔、发布时间、下一章入口同步回 `story_bible`。
- **前端页面**并不自己做推理，它只是不断拉取 `story-studio / live-runtime / tasks` 等接口，把后台运行中的状态展示出来。

## 你可以把它想成三层

第一层是**持久化层**：数据库表里落地的内容。  
第二层是**叙事状态层**：`story_bible` 这个超大 JSON，是全书的运行脑。  
第三层是**生成执行层**：初始化、规划、草稿、校验、摘要、标题精修、事实登记、连载状态刷新。

这套结构的妙处在于：  
它不是一次性生成整本书，而是把小说当成一个长期运行的状态机。每一章不是孤立产物，而是**上一章状态 + 当前规划 + 近因上下文 + 硬事实约束 + 连载策略**共同算出来的结果。
