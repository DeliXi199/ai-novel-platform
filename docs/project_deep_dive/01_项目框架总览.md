# 01 项目框架总览

## 1. 这是什么项目

这是一个“AI 连载小说生产平台”。  
它的目标不是只生成一段文本，而是把小说创作拆成以下几个长期可维护的环节：

- 创建一本小说并初始化设定。
- 生成全书总纲、阶段剧情弧、近期章节卡。
- 按章节顺序生成正文。
- 为每一章登记摘要、角色变化、线索、钩子、硬事实。
- 支持库存模式与即时发布模式。
- 支持人工干预、异步任务、任务重试、任务取消。
- 支持工作台查看全局运行状态。

说得更直白一点：它是“小说生产线”，不是“单次聊天写作工具”。

---

## 2. 顶层目录怎么理解

### `backend/`
后端主体。  
这里面有 FastAPI 接口、SQLAlchemy 模型、Pydantic Schema、章节生成服务、初始化服务、事实守卫、状态同步逻辑、导出/TTS 等功能。

### `frontend/`
前端静态页面。  
没有单独的 Node 服务，也不是复杂前端工程，而是由 `index.html + app.js + 少量模块` 组成的同源页面工作台。用户的大多数操作，其实都是调用后端接口。

### `docs/`
项目已有文档和本次补充的深度拆解文档都在这里。

### `backend/alembic/`
数据库迁移脚本。  
这里定义了表结构如何随着版本演进。

### `backend/tests/`
测试。  
数量很多，说明这个项目的“状态层与生成逻辑”并不是随手拼出来的，它有明显的回归测试意识。

## 3. 后端分层结构

### 3.1 API 层：`backend/app/api/routes/`

这一层负责暴露 HTTP 接口，不负责核心生成算法。  
它做的事情更像是：

- 接收请求参数。
- 校验基本条件。
- 调用 service 层。
- 把异常转成 HTTP 响应。
- 返回给前端结构化 JSON。

主要按“小说”这个资源拆路由：

#### `novel_management.py`
负责小说列表、创建、异步初始化、重试初始化、重命名、删除。

#### `novel_runtime.py`
负责工作台、Story Workspace 快照、任务状态、规划状态、连载模式、实时运行态等查询与调整接口。

#### `novel_chapters.py`
负责章节列表、单章/批量生成、批量发布库存章、删除末尾章节、导出、TTS 等。

#### `novel_interventions.py`
负责人工干预指令的录入与查询。

#### `health.py`
健康检查与 LLM 可用性检查。

---

### 3.2 Schema 层：`backend/app/schemas/`

这一层定义“接口输入输出长什么样”。  
也就是前端和后端传输时的数据外壳。

注意：  
Schema 不等于数据库结构。  
数据库讲的是“怎么存”；Schema 讲的是“怎么收、怎么回”。

---

### 3.3 Model 层：`backend/app/models/`

这一层定义数据库表对象：

- `Novel`
- `Chapter`
- `ChapterSummary`
- `Character`
- `Intervention`
- `AsyncTask`
- `AsyncTaskEvent`

这些是稳定的、会进数据库的主对象。

---

### 3.4 Service 层：`backend/app/services/`

这才是整个项目的大脑。  
这里不是一个单体文件，而是拆得很细，原因很现实：**小说生成不是一步，而是多阶段流水线**。

大致可以按职责理解：

#### A. 初始化与 Story Bible 搭建

- `novel_lifecycle.py`
- `story_architecture.py`
- `story_blueprint_builders.py`
- `story_runtime_support.py`

#### B. 章节生成流水线
- `chapter_generation_entry.py`
- `chapter_generation_prepare.py`
- `chapter_generation_stages.py`
- `chapter_generation_planning.py`
- `chapter_generation_postprocess.py`
- `chapter_generation_persistence.py`
- `chapter_generation_types.py`
- `chapter_planning_support.py`
- `chapter_context_support.py`

#### C. 质量与约束

- `hard_fact_guard*.py`

#### D. LLM 交互与提示词

#### E. 附属能力

## 4. 这个项目最核心的“真正状态中心”是谁

答案不是数据库里的 `Chapter` 表。  
答案是：**`Novel.story_bible`**。

数据库确实保存了“书、章、摘要、角色”等对象。  
但真正驱动小说连续生成的，是 `story_bible` 这个 JSON 大对象。它里面包含：

- 项目卡（project card）
- 世界设定
- 力量体系
- 章节近纲
- 当前剧情弧 active arc
- 下一段剧情弧 pending arc
- Story Workspace 快照
- 规划层状态
- 长期状态层
- 事实账本
- 硬事实守卫状态
- 连载发布状态
- 工作流状态
- 实时运行快照

可以把它理解成：  
数据库是仓库，`story_bible` 是驾驶舱。

---

## 5. 为什么要同时有数据库表和 `story_bible`

因为这两类数据不是一个层级的东西。

### 数据库表适合保存：
- 一本书的基本信息；
- 第 N 章正文；
- 第 N 章摘要；
- 某个人工干预；
- 某个异步任务及其事件日志。

### `story_bible` 适合保存：
- 当前规划到哪一章；
- 哪些伏笔待偿还；
- 最近 7 章的近纲；
- 当前剧情弧与待切换剧情弧；
- 主角长期状态与资源状态；
- 已发布章节的硬事实总账；
- 当前生成流程跑到哪一步；
- 下一章生成时要用的结构化上下文。

如果全丢进数据库表，表会爆炸得像一锅面。  
如果全丢进一个 JSON，又会失去可查询的主对象边界。  
所以这个项目采用的是混合模式：**主实体进表，复杂运行态进 JSON**。

## 6. 从运行方式看，它更像一个“状态机”

用户看到的是“点一下生成下一章”。  
系统真正做的是：

1. 读小说当前状态。
2. 确认是否已有下一章规划。
3. 如有必要自动预取下一段剧情弧。
4. 组装本章执行卡、场景卡、局部上下文。
5. 调 LLM 草拟正文。
6. 做质量检查、标题精修、摘要提炼。
7. 更新事实账本与长期状态。
8. 决定这一章是直接发布还是进入库存。
9. 刷新下一章入口与工作台状态。

这明显不是“生成文本”，而是“推进状态机”。

---

## 7. 前端在整套架构里的位置

前端不是创作引擎，只是控制面板。  
它主要做三件事：

- 发送操作：创建、生成、发布、删除、干预、切换模式。
- 轮询或监听状态：任务进度、live runtime、story-studio。
- 把 Story Bible 里的重要片段以工作台形态展示出来。

所以你看前端，会发现它依赖很多“快照接口”。  
这说明前端不是直接理解复杂生成逻辑，而是等待后端把“已整理好的运行态”喂给它。

---

## 8. 一句话总结架构

这套项目的结构可以概括成：

**FastAPI 接口壳 + SQLAlchemy 主实体 + Story Bible 运行脑 + 多阶段章节生成流水线 + 异步任务系统 + 静态工作台前端。**

它的关键设计思想是：  
**小说不是一次产出，而是持续演化的系统状态。**
